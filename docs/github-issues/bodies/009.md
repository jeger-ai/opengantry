## Problem

`init-asset-catalog.ts` hardcodes IDE asset entries and tags while `templates/integrations/compatibility.json` already declares each integration's `canonical_paths` and `recipe_file`. The `_compat` parameter to `resolveAssetsFromProfile()` is unused (`void _compat` at line 192), so catalog and compat can drift silently — e.g. Cursor ships five `.cursor/*` assets in the catalog but only two paths in compat. Adding an integration today requires edits in four places: `compatibility.json`, `integration-compat.ts`, `init-asset-catalog.ts`, and recipe templates.

## Acceptance

- [ ] Build IDE-pack portion of `INIT_ASSET_CATALOG` from `loadIntegrationCompat()` `canonical_paths` (+ documented override table for non-canonical assets like Cursor hook scripts)
- [ ] Derive `InitAssetTag` IDE keys from `INTEGRATION_IDE_KEYS` (single enum source)
- [ ] `resolveAssetsFromProfile()` uses compat data (remove `void _compat`)
- [ ] Test: every `canonical_paths` entry resolves to a catalog asset; every catalog IDE asset traces to compat or override doc
- [ ] `init-profile.test.ts`, `template-parity.test.ts`, `upgrade.test.ts` still pass
