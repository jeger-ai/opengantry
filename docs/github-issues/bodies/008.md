## Problem

`mcp-runtime.ts` and `mcp-legislation.ts` already use discriminated unions for MCP payloads, but `mcp-orchestration.ts` and `mcp-upgrade.ts` still return `Record<string, unknown>` despite typed sources (`StartResult` in `start-orchestration.ts`, `UpgradePlanResult` / `UpgradeApplyResult` in `upgrade-plan.ts` / `upgrade-apply.ts`). The orchestration handler also adds undeclared fields (`pinned_mission`, `runtime_env`, `resolve`) that are invisible to callers and tests (`start.test.ts:85-106`). Zod validates MCP **inputs** only in `mcp-tools-register.ts` — not outputs.

## Acceptance

- [ ] Export `StartOrchestrationMcpResult` (discriminated `status: "ok" | "failed"`) covering snake_case fields plus optional `pinned_mission`, `runtime_env`, `resolve`
- [ ] Export `UpgradePlanMcpResult` / `UpgradeApplyMcpResult` (or reuse typed plan/apply results + shared error union) from `mcp-upgrade.ts`
- [ ] Replace `Record<string, unknown>` return types; reuse `McpErrorBody` from `mcp-legislation.ts` for error paths
- [ ] Update `start.test.ts` MCP assertions to use typed payloads
- [ ] `npm run validate` / unit tests pass

## Priority

**Low** — post-v1.1.1 tactical debt; no runtime bug, but inconsistent with the typed MCP direction.
