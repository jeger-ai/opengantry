## Problem

Teachers approving substrate upgrades see a file list (`planned_writes[]`, staged hashes) but not **what changed** between `from_version` and `to_version` (behavior, breaking changes, hook/MCP deltas). `upgrade-plan.ts` logs file list only; no root `CHANGELOG` in repo. Increases upgrade friction and surprise regressions for enterprise rollouts.

## Acceptance

- [ ] Ship `CHANGELOG.md` or `templates/CHANGELOG.md` in npm package; `gapman upgrade plan` / `--dry-run` prints relevant section for version range
- [ ] JSON plan includes `changelog_excerpt` or `release_notes_url`
- [ ] MCP `gxt_upgrade_plan` returns same excerpt
- [ ] Fallback when no changelog entry: warn + link to GitHub releases
- [ ] Test: plan from N→N+1 includes non-empty notes when changelog exists

## Priority

**Low** — adoption UX; upgrade path already works atomically.
