## Problem

OpenGantry's "cage" is strongest at subprocess boundaries (`runtime exec`) and weakest where most IDE agents actually edit files (native Write/Edit). MCP two-step legislation protects mission YAML creation but not general repo writes; `gxt_start_orchestration` can still write missions without a draft token when `write_mission=true` (`mcp-orchestration.ts`). Cursor hooks only *ask* on shell paths to law/manifest (`.cursor/hooks/gxt-before-shell.sh`), not deny. `docs/INTEGRATIONS.md` explicitly marks IDE Write/Edit as advisory.

## Acceptance

- [ ] ADR defines containment tiers: what is blocked vs ask vs advisory per host capability (Cursor MCP, shell, Write/Edit)
- [ ] All MCP paths that **write mission or governance files** require draft-token approval (including `gxt_start_orchestration` when `write_mission=true`)
- [ ] New or extended host hook (e.g. `beforeMcpExecution` / file-write hook where supported) denies writes outside effective TMVC or into `forbidden_zones` for pinned missions
- [ ] Fail-closed behavior with stable `GXT_*` error codes and remediation in MCP JSON envelope
- [ ] `docs/INTEGRATIONS.md` enforcement table updated; no claim that IDE edits are "trapped" unless containment is on
- [ ] Tests: MCP write attempt outside TMVC → denied; legislation still works via draft/execute; `runtime exec` behavior unchanged

## Priority

**Major** — v1.2 flagship; core "proactive containment" theme. Requires ADR + Teacher mission before implementation (Tier-3 substrate).
