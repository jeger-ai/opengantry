## Problem

New adopters can complete `gapman onboarding` without valid hooks path, MCP config, Teacher allowlist, or detected IDE wiring. `onboarding.ts` is an interactive walkthrough — **does not run** `gapman doctor` or `doctor-integration-checks.ts`. Integration issues surface late at first verify/push despite doctor probes existing for IDE CLIs.

## Acceptance

- [ ] Onboarding Step 0 (or post-init): run `collectDoctorReport` + `runIntegrationDoctorChecks`; surface blockers before mission loop
- [ ] For selected init profile IDE, assert canonical paths exist (from `compatibility.json`); fail or strong-warn with fix commands
- [ ] Optional `--strict` onboarding exit non-zero on doctor `fail` lines
- [ ] Cursor-specific: verify `.cursor/hooks.json` version + MCP config presence when cursor detected
- [ ] `gapman init --tutorial` reuses same gates
- [ ] Tests in `onboarding.test.ts` / extend `init-profile.test.ts`

## Priority

**Medium** — adoption; reduces support burden; aligns with `doctor-integration-checks.ts`.
