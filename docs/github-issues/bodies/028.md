## Problem

`gapman triage` returns a single reason with no visibility into which rules fired, which skills matched, or confidence. `triage-logic.ts` uses substring + keyword heuristics on `path_risks` and `risk_keywords` — escalates often on ambiguous intents. Teachers cannot audit routing decisions; JSON output lacks explainability fields required by SOUL contract evolution.

## Acceptance

- [ ] Extend `TriageResult` with explainability fields (e.g. `matched_skills`, `risk_hits`, `path_risk_hits`, optional `confidence` or ordered `signals[]`)
- [ ] `--json` and human output list contributing factors (SOUL-aligned, still manifest-only routing)
- [ ] Routing unchanged: binary `action` remains authoritative; scores are **non-binding**
- [ ] Update `SOUL.md` / MCP `gxt_start_orchestration` payloads
- [ ] Tests for multi-match and risk_keyword + skill overlap cases

## Priority

**Medium** — UX/debugging; no security change if routing stays binary.
