# OpenGantry v2.5.0 — Proposed release slice

**Status:** Draft proposal, rev 3 (2026-07-08 — execution-risk review incorporated: Theme C reordered coverage-before-consolidation, C-item IDs deduplicated, B2 locked to binary gate, A2 doctor check locked deterministic, C2 scoped to happy-path I/O lock tests, C3 constrained to atomic chunk PRs)  
**Context:** v2.4.0 shipped the architecture cage; backlog currently jumps to v3.0 adaptive perimeter (#61–#63, #86–#91).

---

## Where the project stands

v2.4.0 shipped the **architecture cage**:

- `TARGET_ARCHITECTURE.yaml` + `gantry arch check` (deterministic boundaries)
- Advisory `ARCHITECTURE_RUBRIC` judge (KPI findings, never flips PASS/FAIL)
- External architecture pointer fetch (`gantry arch fetch`)
- Verify SARIF/JUnit export (`--format sarif|junit`)

The backlog then jumps straight to **v3.0 adaptive perimeter** (#61–#63 auto-discovery / judge / blueprint, #86–#91 defensive profiles). Those are large, Tier-3, ADR-first items.

That leaves a gap a minor release should fill: **the v2.4 cage is dogfood-only, and tactical debt will ossify under v3.0.**

### Key finding

`gantry arch check` only scans TypeScript files under a hardcoded `src/cli/` path (`src/cli/lib/target-architecture.ts`), the schema version is locked at `0.1.0`, and adopters get no `TARGET_ARCHITECTURE.yaml` scaffold from `gantry init`. The headline v2.4 feature works for OpenGantry itself but not yet for adopters.

v3.0's auto-discovery engine (#61) is supposed to *generate* that YAML — generalizing the checker first is a hard prerequisite.

---

## Proposed theme: Adopter-ready cage + defensive guard foundations

v2.5.0 should make the architecture cage **adopter-installable**, land the **schema foundation** for v3.0 defensive profiles, and **pay down tactical debt** before the big v3.0 scanner work.

## Must-have: Defensive guard foundation (Theme B)

Issues [#87](https://github.com/jeger-ai/opengantry/issues/87)–[#91](https://github.com/jeger-ai/opengantry/issues/91) are **deterministic** guards (LOC budget, churn ratio, test-to-code ratio, file-scope blast radius). They belong with the existing deterministic gate stack, not with the LLM-heavy v3.0 items.

Pull the **foundation** into v2.5.0; leave interactive onboarding and remaining guards for v3.0.

| ID | Task | GitHub issue | Rationale | Tier |
|----|------|--------------|-----------|------|
| **B1** | **Defensive profile schema** — `defensive_profile` block in `.gitagent/config.json`, validated, fail-closed defaults | [#87](https://github.com/jeger-ai/opengantry/issues/87) | Pure schema + validation; everything else in #88–#91 depends on it | Tier-3 (ADR + mission) |
| **B2** | **Net LOC budget guard** — first concrete guard wired into `gantry verify` as a phase, reusing the `trusted_automation` `max_net_loc` machinery in `gxt-manifest-lib.mjs` | [#90](https://github.com/jeger-ai/opengantry/issues/90) | Cheapest guard to land; proves profile → verify-phase plumbing before v3.0 adds the rest | Tier-3 |

**B2 scope constraints (locked):**

- **Binary gate only** — the guard emits pass or fail. No warning tiers, no advisory mode, no interactive CLI prompts. All interactive behavior stays in the v3.0 onboarding cycle (#86).
- **Cross-platform pathing** — before recycling `gxt-manifest-lib.mjs`, verify its diff/path handling is correct across runtimes and volume layouts adopters actually use (Windows drive letters, case-insensitive filesystems, git worktrees). Add regression tests for these before wiring the guard into verify.

| [#88](https://github.com/jeger-ai/opengantry/issues/88) | Test-to-code ratio stability guard |
| [#89](https://github.com/jeger-ai/opengantry/issues/89) | Churn ratio guard for rewrite-heavy patches |
| [#91](https://github.com/jeger-ai/opengantry/issues/91) | File scope leakage guard (blast radius limit) |

**Delivery order:** B1 → B2 (ADR first for B1).

## Must-have: Debt and drift pay-down (Theme C)

**Ordering rule (locked): coverage before consolidation.** Consolidating ~95 micro-modules across code paths that lack command-level tests would risk silent behavioral regressions in the deterministic gate stack. C3 MUST NOT start until C2 has landed baseline integration tests pinning the current inputs/outputs of `scan`, `register`, and `check`.

**C2 execution scope (locked):** baseline integration tests here means **happy-path I/O contract locks only** for `scan`, `register`, and `check` (known-good inputs, stable output shape, deterministic exit status). This is not an exhaustive matrix pass.

**C3 execution scope (locked):** consolidation MUST be split into **atomic, domain-specific chunks** (for example: filesystem utilities, parser/mission parsing helpers, verify presentation helpers) across multiple small PRs. A single large consolidation PR is out of scope.

## Suggested delivery order

```
C1                    (immediate maintenance — ship first, independent of the release)
A1 → A3 → A2          (architecture cage generalization)
B1 → B2               (defensive profile foundation; ADR first)
C2 → C3               (coverage backfill BEFORE consolidation — never refactor untested paths)
C4, C5                (independent fill-in missions)
S1–S3                 (only if slice stays on schedule)
R1                    (release close-out)
```

**Rough sizing:** Themes A and B are ~one mission per row (MSN pattern like v2.4). Theme C rows are each a small single mission, except C3 which is a multi-mission refactor gated on C2.

---

## Strategic rationale

v3.0's auto-discovery ([#61](https://github.com/jeger-ai/opengantry/issues/61)) and blueprint ([#63](https://github.com/jeger-ai/opengantry/issues/63)) both *emit* `TARGET_ARCHITECTURE.yaml`. Shipping a generalized, versioned, adopter-installable **consumer** of that format in v2.5.0 means v3.0 lands on solid ground instead of retrofitting the checker mid-flight.

Similarly, landing the defensive profile **schema** (#87) and one concrete guard (#90) in v2.5.0 de-risks the v3.0 profile onboarding (#86) and remaining guards (#88–#91).
